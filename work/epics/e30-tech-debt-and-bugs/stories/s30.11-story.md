# Story S30.11: Protocolo de Extracción Nodal Frontend

## User Story (Connextra Form)
Como **Product Architect**
Quiero **formalizar el Protocolo de Extracción Nodal (`getValue() / getValidatedValue()`) a lo largo de todos los Componentes de Interfaz de Usuario que envuelven su estado interno aisladamente**
Para **resolver la asimetría arquitectónica (H10) donde el motor de recolección principal del `UI_FormSubmitter` lidia con componentes anidados de manera dispar o asumiendo el DOM global.**

## Contexto de Negocio
Durante el escrutinio de los Subgrids y la refactorización arquitectónica de "Passive DOM to Local State" en la historia `S30.3`, emergió un red-flag heurístico: la asimetría procedimental. Componentes como el multi-seleccionador atómico o el relation-builder tienen interfaces no estandarizadas para vaciar la memoria y extraer su información de vuelta hacia el orquestador maestro (Submit). 

## Acceptance Criteria
- **Scenario**: Recolectar datos asíncronamente desde el Submit Controller.
  - **Given** que el usuario hace click en Guardar y `UI_FormSubmitter` clona las directivas a su estado inicial.
  - **When** procede a iterar clases webcomponents (Selectores Multi, Relacionales Bidireccionales).
  - **Then** el Submitter utiliza implícitamente un método Duck-Typed como `.getValidatedValue()` forzosamente alojado en el wrapper DOM de cada subformulario/componente, anulando los *Hacks RegExp* para deducir qué estado alberga.
  
- **Scenario**: Reducción de lógica frágil en FormSubmitter.
  - **Given** que la interfaz posee 3 componentes dinámicos pesados distintos.
  - **When** se levanta la serialización JSON del formulario.
  - **Then** el `FormSubmitter` ya no necesita contener un mega `switch-case` asimétrico por cada componente, delegando esa extracción nodular a las clases contenedoras originales.
