# Scope S30.12: Optimistic UI Server Latency Mitigation & FormEngine Lifecycle

## Boundaries

**In scope:**
- Examinar la creación, mutación y disolución asíncrona de los elementos estéticos `<ion-loading>` e interfaces bloqueantes del framework en `UI_FormSubmitter.client.js`.
- Refactorizar flujos asimétricos fallidos de promesas `catch` u objetos `Response` con Status Inconsistentes desde el Backend (Errores HTTP 500 ocultos en objetos).
- Crear un `Timeout Wrapper` estricto en la resolución de `DataAPI.call` o `Universal Router` para que un fallo en la red (o un loop ciego de AppsScript) intercepte y libere la UI frontal al cabo de X milisegundos evitando el "freeze".
- Arreglar test fallido `DataGrid_Integration` relacionado con UI_FormUtils si requiere un inyector simple para que las integraciones Vitest sanen globalmente.

**Out of scope:**
- Re-escribir el Backend de LockService en AppsScript (Esto ya está mitigado estructuralmente, aquí solventamos el trato visual front-end cuando el back tarda o expira).
- Cambiar librerías de componentes (Ionic).
- Modificar esquemas lógicos de DB.

## Done Criteria
1. Un Guardado fallido (ej. simulación de Timeout AppScript > 20 segundos) jamás debe dejar al usuario atrapado en el Backdrop del DOM. Se debe invocar `_revertButtonState()` e `ion-loading.dismiss()` incluso ante _Unhandled Rejections_.
2. Un Timeout Explícito es gobernado por cliente dentro de `UI_FormSubmitter` o `DataAPI`, alertando visiblemente que la red está congestionada.
3. El Spinner nativo de Ionic en `UI_FormSubmitter` cierra correctamente todos sus children clonados restaurando el Layout del botón a su exacto origen.
