# Story: S30.12 - Optimistic UI Server Latency Mitigation & FormEngine Lifecycle Synchronization

> **RaiSE Framework**: Story Scope Definition

## Objective
Resolver las fricciones de latencia transaccional y visual identificadas por los análisis de estrés End-to-End, mediante la implementación sistemática de patrones de estado asíncrono (Optimistic UI) para bases de datos Cloud lentas (Google Sheets) y la sincronización estricta del ciclo de vida del *FormEngine*.

## Rationale (Why)
Las pruebas automatizadas de Chromium (E2E) expusieron que las secuencias de modelado profundo (ej. Unidad -> Portafolio -> Grupo) sufren embotellamientos no por bloqueos visuales pasivos, sino por latencia RPC del servidor (+5 a 15 segundos para resoluciones multi-capa en GAS). 
Además, la aparición instintiva del DOM choca contra un grafo de metadata (`Logic_Graph_Rules`) que se carga y evalúa pasivamente. Si forzamos al usuario (o al robot automatizado) a esperar descargas íntegras de servidor antes de continuar operando, perpetuamos una "UI Perceptualmente Lenta". La adopción de Optimistic UI inyectará extrema robustez al evitar bloqueos de interacción en operaciones de red (DataStore).

## Requirements

### 1. Optimistic UI en el DataStore_Engine
- Refactorizar las inserciones y mutaciones para que el Front-End (Dashboard y Grid) visualmente re-dibuje los resultados de mutación en memoria *inmediatamente* asumiendo éxito transaccional.
- Enviar los payloads en modo *Background Queue* hacia Google Apps Script (RPC).
- Aplicar estrategias de "*Rollback* Visual" sólo si el puente arroja un error asíncrono, emitiendo un WebComponent Exception Toast.

### 2. Sincronización Transaccional (`APP::READY` / `FORM_READY`)
- `FormEngine_UI.client.js` no debe despachar ni habilitar el estado interactivo (`btn-create-new` ni Modales en Sidebars) hasta que suscriba explícitamente a la recepción del esquema lógico de grafos.
- Garantizar que "la capa semántica" ha concluido su inyección LocalStorage/Cache antes de despertar el EventBus para la Vista.

## Success Criteria
- [ ] Un usuario puede realizar múltiples operaciones CRUD (múltiples n-tuplas) a través de `DataStore` sin verse bloqueado por "Spinners" de Apps Script que duren más de 200ms.
- [ ] La Prueba E2E *"Top-Down Hierarchy Stress"* atraviesa limpiamente las creaciones multicapas dentro de su timeout estricto de `< 5_000ms`, confirmando que el Frontend reacciona a una capa de memoria caché optimista.
- [ ] `FormEngine` nunca dibuja un Action Button activo (`Guardar`) antes de que el grafo `Logic_Graph_Rules` reporte bandera verde (IsHydrated).

## Architectural Considerations
- **SCD-2 & Primary Keys**: La refactorización hacia *Optimistic UI* puede chocar con las reglas de Topología Temp (SCD-2) si no se inyectan ID correlacionales locales. Es mandatorio que la UI asigne Uuids Sintéticos en `Adapter_Sheets.client.js` mientras la API despacha su Primary Key definitiva.

## Dependencies
- **S30.7**: Estandarización Universal de Primary Key completada, lo cual proveerá la confianza estructural de inserciones optimistas.
