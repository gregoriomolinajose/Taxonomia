# Story S30.8: Autogenerador Lexical (Jira-style)

## User Story
**Como** arquitecto de datos del sistema,
**Quiero** que el orquestador backend asigne un ID universal silencioso (UUID puro) para la integridad profunda de persistencia Y además construya un ID visual amigable (ej. `PRTF-4`) único y secuencial para los usuarios,
**Para** garantizar integridades de bases de datos de alto rango mientras mantenemos una experiencia de usuario corporativa limpia y fácil de memorizar.

## Context
Tras acotar la Estandarización de las Llaves Primarias (S30.7), nos dimos cuenta que mezclar Acrónimos Puros (ej. `PORT-`) dentro de los UUID generados aleatoriamente puede sacrificar eficiencia e indexado en algunas BD y ensucia el identificador del usuario. El estándar Jira separa un Unique ID de backend de un Ticket ID front. Queremos eso: Las llaves backend deben ser puras, las visuales deben ser fáciles.

## Criterios de Aceptación (Gherkin)

**Scenario: Implicit Primary Key Integrity**
Given se invoca `API_Universal` para crear una entidad *carente* de una definición PK obligada
When el backend procesa el Schema
Then el adaptador debe forzar la inyección de una columna `id` (por defecto purista) garantizando el salvamento idempotente sin requerir la intervención del desarrollador.

**Scenario: Generation of Lexical Identity**
Given yo inserto una Unidad de Negocio nueva de nombre "Alpha"
When la persistencia emite el Commit Final
Then el Schema dicta que el prefijo es `UNDN` y la base de datos se encarga de asignar el `UNDN-1` de manera consecutiva (si es la primera entidad).

**Scenario: Form Presentation (Badge)**
Given abro la entidad `UNDN-1` en mi Dashboard
When se cargan los componentes web nativos del form builder
Then la Identidad (`UNDN-1`) aparece como un Badge estético en la parte superior izquierda y NO se muestra ninguna caja de Input editable que diga "id unida negocio", para garantizar la vista ejecutiva.
