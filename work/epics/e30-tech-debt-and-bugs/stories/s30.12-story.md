# Story S30.12: Optimistic UI Server Latency Mitigation & FormEngine Lifecycle Synchronization

## User Story (Connextra Form)
Como **Product Architect**
Quiero **Sincronizar el ciclo de vida transaccional del FormEngine introduciendo Mitigación de Latencia en UI**
Para **Prevenir los bloqueos visuales perpetuos (spinners atascados) que asedian la plataforma durante cuellos de botella de red hacia AppScript, mejorando radicalmente el UX ante la lentitud del backend.**

## Contexto de Negocio
Taxonomia opera bajo la arquitectura REST/RPC de Google Apps Script. Durante momentos de estrés de Google u operaciones masivas (Topological Save), las inserciones pueden tardar varios segundos, y en ciertos escenarios de fallback fallos catastróficos u "Optimistic Concurrency Exceptions", la UI visual deja bloqueado el flujo (Spinners ciegos, modales que no cierran). Es urgente consolidar una UI Defensiva (Optimista) que actúe inmediatamente y libere al usuario mientras el "Worker" interactúa en segundo plano o que provea Fallbacks de restauración visual efectivos ante errores no controlados.

## Acceptance Criteria
- **Scenario**: Transacción de Red en segundo plano
  - **Given** Un usuario somete un subformulario denso
  - **When** El servidor de GCP entra a LockService por N segundos.
  - **Then** La arquitectura UI_FormSubmitter o FormEngine encapsula el Spinner de "Guardando..." en un ciclo controlado (Timebox) y en caso extremo de colapso, resucita la Interfaz revirtiendo el estado estético sin colgar jamás el navegador y mitigando la desesperación del usuario brindando "Optimistic pre-saves" o Dismiss forzosos con errores claros.
  
- **Scenario**: FormEngine Lifecycle Hooks
  - **Given** El Guardado finaliza con un DispatchEvent ('FormEngine::InlinePersisted').
  - **When** La operación no lanza error local.
  - **Then** La Modal u origen visual asíncrono se disuelve impecablemente resolviendo promesas sin dejar overlays fantasmas (backdrop issues).
