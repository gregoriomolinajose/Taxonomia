## Heutagogical Checkpoint
1. **What did you learn?**  
   Que el manejo de hojas de estilo en entornos encapsulados modernos (Shadow DOM, Ionic WebComponents, anidados jerárquicos de CSS variables) penaliza fuertemente el uso de motores como `clean-css` (los cuales heredan parsings rápidos pero frágiles dependientes de REGEX). La inserción de `esbuild` por ser un generador AST completo garantiza el respeto absoluto a la topología original de las agrupaciones.
2. **What would you change about the process?**  
   Auditar proactivamente el pipeline CLI de NodeJS de manera más periódica dado que mutar archivos de compilación acarrea consecuencias a todos los dominios de negocio.
3. **Are there improvements for the framework?**  
   Unificar (en un futuro Epic) los empacadores y compiladores del JS cliente y HTML bajo el mismo paraguas `esbuild` en lugar de delegar CSS a Esbuild y JavaScript a manipulación sincrónica string.
4. **What are you more capable of now?**  
   Entender la diferencia computacional y de resiliencia entre la sustitución léxica (regex) y los Parseos Estructurales (AST).

## Improvements Applied
- Desinstalado nativamente `clean-css`.
- Instalado `esbuild`.
- Modificada llamada `const output = new CleanCSS().minify...` por el robusto `esbuild.transformSync()`.

## Patterns Added/Reinforced
1. **AST Tooling over Regex Pipeline:** Siempre delegar mutaciones complejas (como minificar y agrupar lenguajes foráneos) a un analizador AST `(Pattern)` en lugar de manipulaciones en crudo de JS, validando el postulado "Configuration over Convention".
