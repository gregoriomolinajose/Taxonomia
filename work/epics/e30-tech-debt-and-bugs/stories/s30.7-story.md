# Story S30.7: Estandarización Universal de Primary Key

## User Story
**Como** arquitecto de datos del sistema,
**Quiero** establecer una declaración explícita de clave primaria (`primaryKey`) en el motor de jerarquías para cada tabla o entidad central del ecosistema,
**Para** garantizar que los adaptadores de base de datos (`Adapter_CloudDB`, `API_Universal`, etc.) no requieran depender de inferencias arriesgadas o fallos silenciosos durante la mutación de estado.

## Context
Actualmente algunas tablas definen cuál es su clave primaria empíricamente porque el código asume que el primer campo indexable es "id". Queremos asentar la gobernanza donde el Schema Diccionary dicte ineludiblemente quién gobierna a la entidad para que el ORM actúe acorde sin hardcoding de parámetros.

## Criterios de Aceptación (Gherkin)

**Scenario: Explicit Primary Key in Schema**
Given el archivo `Schema_Engine.gs` define la configuración
When se declara una nueva tabla como entidad principal
Then la sintaxis debe contemplar el atributo de raíz `primaryKey: 'id'` (o alternativo) explícitamente y visible para cualquier engine consultor.

**Scenario: ORM Safe Fallback**
Given el `Adapter_CloudDB` intenta inicializar un UPDATE sobre un registro
When consulta al `APP_SCHEMAS.Entidad`
Then el código intercepta la respuesta leyendo `.primaryKey` o por descarte forzado de sistema `.id` para no colapsar.
