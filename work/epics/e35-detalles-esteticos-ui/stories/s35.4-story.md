# Story S35.4: Centralización de Reglas Topológicas y Abstracción de UI Patterns

## User Story (Connextra)

**Como** desarrollador de infraestructura UI
**Quiero** centralizar la lógica de filtrado jerárquico y los patrones de seguridad de foco (Blur Race Condition)
**Para** eliminar la duplicación técnica detectada en el Architecture Review (H9, H10) y asegurar que las reglas de negocio jerárquicas sean consistentes en todos los componentes.

## Acceptance Criteria (Gherkin)

**Scenario 1: Centralización de Filtros Jerárquicos**
**Given** un dataset de entidades con niveles (nivel_tipo)
**When** se aplican reglas de topología (`strictLevelJumps`, `levelFiltering`)
**Then** una única función en `UI_FormUtils` debe devolver el dataset filtrado, eliminando la lógica duplicada en `RelationBuilder`.

**Scenario 2: Robustez de Foco (Blur Guard)**
**Given** un selector SearchableSingle en Desktop
**When** un usuario hace clic en una opción de la lista
**Then** el flag de protección contra el desenfoque prematuro debe estar debidamente documentado o abstraído para prevenir regresiones UX.

## Technical Tasks
- [ ] Implementar `UI_FormUtils.filterByTopology(dataset, rules, currentLevel, relationType)`
- [ ] Refactorizar `RelationBuilder` para usar el nuevo helper (eliminando duplicación en init y reload).
- [ ] Verificar que las reglas de "Padre no permitido en Nivel 1" (rootRequiresNoParent) sigan funcionando.
- [ ] Realizar prueba de regresión con la suite QA inyectada.
