# S38.6: Cobertura de Pruebas Integrales para el Hub de Ingesta Inteligente

## Problem & Value
**Problem**: El Hub de Ingesta es el corazón transaccional para operaciones en masa. Sus funciones involucran divisiones de arrays pesados, filtrado invisible, resolución de identidades lógica en `O(1)`, e intercepción en el pipeline de Requests de `API_Universal`. Cualquier refactor futuro podría romper sus axiomas e introducir corrupciones silenciosas (ej: inyectar el mismo usuario 5 veces o ahogar a Apps Script por sobrepasar el TimeOut iterativo).
**Value**: Garantizar un determinismo a largo plazo al someter las piezas críticas a automatización. Asegura el principio de Fail-Open de Workspace y el "Single Responsibility" de `Engine_ETL.gs`. 

## Approach
Crearemos 3 tipos de suites testando la Arquitectura Modular.

1. **Unit Tests (Backend `Engine_ETL.gs`)**: 
   Aislar la función `hydrateAndDeduplicate` en Jest mediante el archivo `tests/backend/Engine_ETL.test.js`. Al no estar directamente expuesta a Node (por su diseño de Google Apps script) será evaluada importando su contexto y mockeando variables maestras como `APP_SCHEMAS`, `resolverDirectorioWorkspace` y devolviendo un Mock de memoria de 3 filas persistidas por `Engine_DB.list`.
2. **Integration Tests (Backend `API_Universal.gs`)**:
   Probar que si se dispara el método maestro `API_Universal.bulkInsert(entity, data)`, se encadena exactamente un `_generateShortUUID`, pasándolo por el Spy() de `Engine_ETL.hydrateAndDeduplicate` y finalmente entregando a `Engine_DB.upsertBatch`.
3. **Frontend E2E / Unit (Vitest `DataEngine_ETL.client.js`)**:
   Inyectar 105 registros ficticios en la variable purgada de un frontend simulado (mediante Happy DOM o jsdom), validando que se emitan tres llamadas (2 de 50 elementos y otra remanente de 5) contra Google mediante `DataAPI.call`.

## Examples
### Jest: Backend Deduplication Pass-through
```javascript
// Dado: Base de datos posee Juan en la llave 'email': juan@coppel.com
const dbRows = [{ id_persona: "A1", email: "juan@coppel.com" }];
const newBatch = [
   { email: "juan@coppel.com", puesto: "Lider" }, // Mismo correo, no trae ID
   { email: "pedro@coppel.com", puesto: "Dev" }   // Nuevo
];

Engine_ETL.hydrateAndDeduplicate('Persona', newBatch);

// Resultado Esperado:
expect(newBatch[0].id_persona).toBe("A1"); // Match y retención O(1)
expect(newBatch[1].id_persona).toBeUndefined(); // Pasa limpio
```

## Acceptance Criteria
- **MUST**: Utilizar Jest para las pruebas de Lógica Transaccional (GAS / Backend), requiriendo de Mocks de ambiente inyectados, evaluando un branch coverage sobre `hydrateAndDeduplicate`.
- **MUST**: Que las pruebas documenten escenarios críticos (ej: "Sábana parcial sin nombre", "API Rate-Limit caida").
- **MUST**: Emplear Vitest para el algoritmo Frontend del `_dispatchChunks` asegurando la limpieza anti-corrupción (borrado de _id, sys_, logs en el import).
