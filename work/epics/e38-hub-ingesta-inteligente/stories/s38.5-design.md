# S38.5: Hardening de DB Batch y Re-hidratación Automática para Persona (Workspace Lookup)

## Problem & Value
**Problem**: Actualmente, la carga masiva de datos (bulkInsert) a través de CSV no sabe resolver entidades cuando falta la llave primaria (`id_persona`). Esto ocasiona que si se inserta el mismo empleado múltiples veces solo con su correo, se le asignen UUIDs aleatorios generando copias exactas y huérfanas en la base de datos.
**Value**: Mejorar la integridad y consistencia del Hub de Ingesta al habilitar a las operaciones de Bulk para operar mediante deduplicación pasiva (Identity Resolution) basada en llaves únicas de dominio, asegurando además que se auto-rehidraten para mejorar la UX y experiencia administrativa.

## Approach
Implementar un *Batch Interceptor* en la etapa temprana de `Engine_DB.upsertBatch`.
1. **Deduplicación Lógica**: Antes de pasar los datos crudos a `_Adapter_Sheets.upsertBatch`, validaremos en `APP_SCHEMAS[tableName]` si la tabla tiene campos donde `unique: true`. De ser así, se obtendrán los registros previos en memoria. Si un campo como `email` coindice entre el payload insertado y el registro en la DB, sustituiremos el primaryKey generado temporalmente por el primaryKey real de la BD, transformando de facto el escenario en un UPDATE transaccional.
2. **Re-hidratación Contextual**: Si detectamos la entidad `Persona`, el sistema validará si le falta el atributo `nombre`. De ser este caso positivo, se conectará en tiempo real (si está configurada la integración de red) a `resolverDirectorioWorkspace` para adquirir e inyectar `avatar`, `nombre`, `apellidos`, `cargo`, y `lider_directo`.

> **Value Preservation Gate**: Se mantiene el modelo de separación de responsabilidades intacto: `API_Universal` (Routing), `Engine_DB` (Logic & Resolution), `_Adapter_Sheets` (I/O plano). 

## Examples

### 1. Ingesta Sábana Incompleta (Automática Re-hidratación)
Si enviamos al servidor un arreglo `[{ email: "jorge.ruiz@bancoppel.com", esquema: "Interno" }]`:
El motor reconocerá que falta su `nombre`. Consultará vía `resolverDirectorioWorkspace` e inyectará:
```json
{
   "email": "jorge.ruiz@bancoppel.com",
   "esquema": "Interno",
   "nombre": "Jorge Alfonso",
   "apellidos": "Ruiz Pineda",
   "avatar": "https://img...",
   "cargo": "Fullstack Lead",
   "id_persona": "PERS-93JS9"
}
```

### 2. Idempotencia y Deduplicación al Subir Dos Veces
Dado que nuestra DB tiene alojado a `jorge.ruiz@bancoppel.com` en su línea actual (`id_persona: "PERS-93JS9"`). Si un analista de datos re-sube el CSV por error horas después pero esta vez adjuntándole `numero_empleado: 99119933` pero careciendo del `id_persona`.
El motor cruzará `payload.email === dbRow.email`, descartará el nuevo UUID asignado temporalmente de fábrica, y aplicará: `payload.id_persona = "PERS-93JS9"`.

## Acceptance Criteria
- **MUST**: Interceptar e identificar llaves secundarias (email) para realizar UPDATE masivos en vez de CREATE masivos, respetando la Regla C-01.
- **MUST**: Usar la API Workspace `resolverDirectorioWorkspace` si la entidad se sube con campos requeridos en blanco (ej. `nombre`)
- **MUST NOT**: Romper el contrato asíncrono ni invadir los responsabilidades del Adapter (`_Adapter_Sheets`). La lógica interviene estrictamente en el Engine_DB.
