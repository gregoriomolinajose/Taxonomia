# S38.3: Hub ETL Extractor Híbrido - Retrospective

## Context
- **Epic**: E38 - Hub de Ingesta Inteligente
- **Story**: S38.3
- **Completed**: 2026-04-15

## What Went Well
- **Zero-Touch Extract:** La extracción de datos masivos usando la API de Google Drive/SpreadsheetApp (`SpreadsheetApp.openById`) funcionó sin problemas bloqueantes, probando la interoperabilidad del ecosistema Workspace con Taxonomía.
- **Seguridad ABAC Dinámica:** Se implementó rápidamente la sugerencia de exigir permisos compuestos (`create` y `update`) para blindar el túnel ETL ante escalamiento de privilegios, validando la extensibilidad de `API_Universal`.
- **Ruteo Desacoplado:** El uso persistente de `window.DataAPI.call` simplificó enormemente el intercambio frontend-backend.

## What Could Be Improved
- **Robustez del Loader Modal:** El componente UI `ion-loading` demostró carecer nativamente de bloques try-catch inherentes, requiriendo un refactor durante la revisión de código de Calidad para no generar cuellos de botella fatales de "Spinners Infinitos".
- **Limpieza de Caché y Memoria RAM:** Actualmente el Frontend guarda la súper Sábana en memoria global (`window._etlPayloadCache`). Deberemos ser cuidadosos en S38.4 de liberar (Garbage Collect) ese array en RAM una vez procesado el chunk en DOM para no asfixiar a los clientes móviles.

## Learnings (Knowledge Graph updates)
- **API Universal Security:** Los servicios de ingesta masiva deben ser protegidos con los verbos reales que el despliegue final ejercerá (`create` y `update`) y no solamente el verbo inofensivo de transferencia inicial (`read`).
- **Google Sheets O(n):** Descartar las filas iteradas localmente en el backend (`isEmptyRow`) resulta fundamental para no enviar paquetes IPC (`google.script.run`) hinchados de arrays nulificados.

## Action Items
- Para S38.4 (Chunker UI): Implementar lógica explícita de `window._etlPayloadCache = null;` inmediatamente después del procesamiento DOM para evitar Memory Leaks en el Tab del Navegador.
