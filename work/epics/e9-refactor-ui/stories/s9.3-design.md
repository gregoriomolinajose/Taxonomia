# Story Design: S9.3 - FormEngine DRY Modularization

## Architectural Context
Actualmente, \`FormEngine_UI.html\` dibuja jerarquías en el DOM usando un inmenso bucle generador. Ciertas funciones, como el filtrado de niveles y la inyección de \`<ion-select-option>\`, se declaran y repiten directamente dentro del \`switch (field.type)\`. 
Adicionalmente, los Listeners asíncronos (como el evento custom \`levelChanged\`) están acoplados anónimamente dentro del componente, incrementando el esfuerzo cognitivo (Fat File Smell / Ticket 5) y replicando la rutina de pintado (DRY Violation / Ticket 6).

## Proposed Design
1. **DRY Render Subroutine**
   - Extraer la inyección de etiquetas de opción a una función puramente utilitaria: \`window._populateSelectOptions(selectElement, dataArray, fieldConfig, emptyOptionNode = null)\`.
   - Esta función limpiará el dropdown e inyectará los Nodos \`ion-select-option\` utilizando un map iterativo, abstrayendo la lógica \`document.createElement\` fuera de la mirada procedimental de la UI.

2. **Decoupled Event Controllers**
   - Extraer el bloque anónimo de \`inputEl.addEventListener('levelChanged', ...)\` hacia una sub-rutina segregada superior: \`window._bindLevelChangeRepaint(inputEl, targetSelect, activeData, fieldDef)\`.
   - El generador maestro de componentes solo tendrá que llamar a \`_bindLevelChangeRepaint(...)\`, ahorrando +40 líneas de callback en línea dentro del propio *switchcase*.

3. **Inerción Pura**
   - El Refactoring será completamente *Backwards Compatible* al no remover funcionalidades lógicas, sino puramente reorgnizar el AST de ejecución (Abstract Syntax Tree) hacia métodos separados, aligerando la carga procedimental de `FormEngine.renderForm()`.
