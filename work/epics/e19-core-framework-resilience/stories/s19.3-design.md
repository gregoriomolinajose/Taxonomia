# Story Design: S19.3 (AppEventBus Telemetry Foundation)

## What & Why
- **Problem**: We need a non-blocking mechanism to track user activity, systemic events, and soft errors.
- **Value**: Establishes the foundational channel for tracking application health and usage metrics without impacting the UI render loop.

## Approach
Through an explicit **Architecture Review** of `JS_Core.html`, it was discovered that a primitive `window.AppEventBus` already exists from Epic 11. Instead of building a secondary Pub/Sub architecture, `window.Telemetry` will act as a structural wrapper that:
1. Standardizes payloads.
2. Emits beautiful, colorized local console logs (`console.info`, `console.error`) according to the severity level.
3. Fires `window.AppEventBus.publish('TELEMETRY::TRACK', packet)` so future hooks (like Google Analytics syncs) can subscribe and batch-send them.

- **Components Modified**: `src/JS_Core.html`

## Example (Runnable Contract)
```javascript
// Invocation Example
window.Telemetry.track('API_FAIL', { endpoint: '/roles', status: 403 }, 'ERROR');

// Internal Output Expected Structure
const packet = {
    timestamp: '2026-04-02T15:00:00.000Z',
    level: 'ERROR',
    action: 'API_FAIL',
    payload: { endpoint: '/roles', status: 403 }
};
```

## Acceptance Criteria
- [x] Objeto `window.Telemetry` initialize inside `JS_Core.html`.
- [x] Supports levels: INFO, WARN, ERROR, FATAL, METRIC.
- [x] Fails safe if `window.AppEventBus` is undefined. 

## Technical Risk
None directly. However, we must ensure developers do not pass massive cyclical DOM objects into `payload` which would choke memory.
