# Story Design: S10.3 - Memory Profiling & Defensive E2E

## Architectural Context
Al no disponer de flujos completos en Cypress / Selenium interconectados a Apps Script nativo, el QA en V8 Runtime es ciego. El Motor UI instancia +150 elementos HTML dinámicamente (`document.createElement`) por cada Formulario que el usuario abre, incrustando cientos de listeners de eventos. Con el tiempo (al anidar y cerrar registros maestros), el *Document Object Model* podría inflarse silenciosamente, causando *Thrashing* del CPU al navegador si hay "Memory Leaks" por variables capturadas (Closures) en el Garbage Collector (Ticket 9/12).

## Proposed Design
1. **Garbage Collection Aggressive Teardown (Defensive)**
   Modificaremos el bloque `closeTop()` del `ModalStackController`. Actualmente llama a `topModal.remove()`. Lo reforzaremos introduciendo un desmantelamiento profundo: `topModal.innerHTML = '';` y anulación de cualquier `__onSaveSuccessFallback` para asistir expresamente al Recolector de Basura.
2. **Stress E2E Profiler Tool**
   Construiremos un diagnosticador asíncrono (Tester Script) inyectado como un helper en local (`window.__runMemoryProfile_E2E`). Hará un bucle `for` de N iteraciones que llamará a la API visual simulando clicks rápidos. Contará el número exacto del Baseline (`document.getElementsByTagName("*").length`).
   Al final del bucle, emitirá una radiografía tabular de *Leaks* con `console.table()`.
