# S5.6: Refactor Frontend DAG Agnosticism

## Problem & Value
**Problema:** El Formulario de `Dominio` arrastra los campos obsoletos `orden_path` y `path_completo_es` procedentes de la antigua topología plana (flat tree). También utiliza una lista desplegable 1:N rígida (`id_nodo_padre`) que no representa la capacidad de Grafo N:M adquirida en E5.
**Valor:** Al erradicar los rastros lógicos del árbol plano, independizamos completamente la Entidad Core de su metadata posicional/jerárquica. Otorgar una UI para administrar relaciones polimórficas (Múltiples relaciones) destapa el verdadero potencial del DAG.

## Proposed Changes
1. **Schema_Engine.gs**:
   - `[DELETE]` `orden_path`, `path_completo_es`, `id_nodo_padre` estático.
   - `[NEW]` Campo `dynamic_list` llamado `relaciones_grafo` dentro de `Dominio`, permitiendo agregar iterativamente padres y asignar `tipo_relacion`.
2. **Engine_DB.js**:
   - `[MODIFY]` `orchestrateNestedSave` -> Refactorizar el "Hook" de `_updateGraphEdges` (que solo absorbía `transientParentId` simple) para aceptar un array dinámico proveniente del Formulario e insertarlo transaccionalmente vía SCD-2 en `Relacion_Dominios`.
3. **Frontend Hydration**:
   - Al editar un `Dominio`, `readFull()` ya extrae información anidada para `type: 'relation'`. Si usamos un `dynamic_list`, se debe hidratar el flat property con un prefiltro SCD-2 para repoblar la lista UI dinámicamente.

## Constraints & Gates
- **Value Preservation Gate:** Al remover `orden_path`, la ordenación visual dependerá del renderizador (Treemap o DataView Sorting nativo). Las vistas deben ser estables sin él.
- **Data Mutations:** Sustituir la escritura escalar por Batch SCD-2 Insert en `_updateGraphEdges` es vital para no detonar fallos contables.

## Acceptance Criteria
- APP_SCHEMAS.Dominio omiten las propiedades espaciales heredadas.
- UI permite añadir "N" relaciones de padre con tipos específicos.
- Node backend caduca aristas huérfanas omitidas al guardar (SCD-2) y crea las nuevas.
