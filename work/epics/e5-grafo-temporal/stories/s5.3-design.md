# Story Design: S5.3 - Backend Setter DAG e Inserciones SCD-2

## 1. What & Why
**Problem:** La interfaz envía un objeto JSON plano (ej. `{ nombre: "DomA", id_nodo_padre: "DOM-1" }`). La base de datos, post-S5.2, ya no admite la estructura plana; consta de la tabla estática `Dominios` y la dimensión histórica `Relacion_Dominios`.
**Value:** Transcodificar inteligentemente el Payload dentro de `Engine_DB.js` para efectuar una operación *Transaccional Multi-Hoja* (Actualizar/Crear Dominio, Anular Arista Antigua, Inyectar Arista Nueva).

## 2. Approach
- **Interceptor en `upsertRecord()`:** 
  1. Si `entity === "Dominio"` y el frontend envía `id_nodo_padre` en la petición.
  2. Extraer `id_nodo_padre` y borrarlo del payload entrante para no poluir la tabla principal.
  3. Ejecutar inserción normal `Dominios`. (Obtenemos el PK crudo `id_dominio` de la inserción o actualización).
  4. Interrogar `Relacion_Dominios` por la Arista actual del `Dominio` hijo (O(N) por ahora o usar `find()`).
  5. Si el `Padre` entrante difiere del `Padre` actual:
     - Mutar Arista Actual: `valido_hasta = sysDate`, `es_version_actual = FALSE`. Push via `setValues` o sobreescribir fila.
     - Crear Arista Nueva: `valido_desde = sysDate`, `id_nodo_padre = NuevoPadre`, `es_version_actual = TRUE`. Expandir hoja.

- **Componentes:** `src/Engine_DB.js` (Específicamente las funciones que envían y escriben los datos como `Adapter_Sheets.upsertRow`).
*(Patrón a implementar: Módulo de Orquestación o Sub-Rutina `_handleGraphRelations`)*.

## 4. Acceptance Criteria
- **MUST:** Eliminar el atributo transitorio de topología del objeto principal antes del Write, para mantener la tabla limpia.
- **MUST:** Ejecutar ciclo de Invalidación correctamente: Si el dominio cambia de Padre A a Padre B, la tabla `Relacion_Dominios` debe reflejar la caducidad (SCD Type 2).
- **SHOULD:** Aislar la operación en una subrutina privada para no entorpecer `upsertRecord` genérico.
