# Story Design: S5.1 - Diseño e Inserción de Metadatos Puente

## 1. What & Why
**Problem:** Actualmente la Base de Datos front-facing no soporta la capacidad de administrar, visualizar ni recolectar relaciones jerárquicas en forma de Grafo (múltiples padres, históricas, matriciales). 
**Value:** Inyectando el esquema `Relacion_Dominios` en `JS_Schemas_Config.html`, dotaremos al frontend (DataView/FormEngine) de la directiva exacta de metadatos (Zero-Touch HTML) para que el Backend pueda operar sobre la estructura SCD-2 sin modificar ni una sola línea de código estático.

## 2. Approach
La historia implementará modificaciones estrictamente en los Archivos de Configuración SSOT:
- **`src/JS_Schemas_Config.html` (Modify):** 
  - Declararemos el diccionario `APP_SCHEMAS.Relacion_Dominios` conteniendo los campos para: `id_nodo_padre`, `id_nodo_hijo` y `tipo_relacion`, valiéndose de `lookupSource: "getDominioOptions"`.
  - Agregaremos las banderas temporales SCD-2: `valido_desde`, `valido_hasta`, `es_version_actual`.
  - Removeremos físicamente la declaración del Campo `id_dominio_padre` desde `APP_SCHEMAS.Dominio`.
- **`src/Global_Config.js` (Modify):**
  - Añadiremos los arrays vacíos de inicialización al `window.__APP_CACHE__` o a los catálogos globales `ENTITY_META`.

*Nota de Dominio:* Al destruir `id_dominio_padre` de `Dominios`, todas las funciones UI legacy dependerán de S5.5 (Reconstrucción del Treemap) para volver a graficar datos, pero aseguramos prevención de uso obsoleto.

## 3. Concrete Examples
### Estructura de Salida Esperada (`JS_Schemas_Config` snippet):
```javascript
Relacion_Dominios: {
    primaryKey: "id_relacion",
    titleField: "tipo_relacion",
    fields: [
        { name: "id_nodo_padre", type: "select", width: 6, lookupSource: "getDominioOptions", required: true, section: "Relación" },
        { name: "id_nodo_hijo", type: "select", width: 6, lookupSource: "getDominioOptions", required: true, section: "Relación" },
        { name: "tipo_relacion", type: "select", width: 6, options: ["Militar_Directa", "Matricial_Secundaria", "Colaborativa_Red"], required: true, section: "Atributos" },
        { name: "peso_influencia", type: "number", width: 6, required: true, section: "Atributos" },
        { name: "valido_desde", type: "date", width: 4, section: "Trazabilidad" },
        { name: "valido_hasta", type: "date", width: 4, section: "Trazabilidad" },
        { name: "es_version_actual", type: "checkbox", width: 4, section: "Trazabilidad" }
    ]
}
```

## 4. Acceptance Criteria
- **MUST:** El motor de inicialización carga y propaga la existencia de la tabla al Frontend y Backend.
- **MUST:** La forma visual (FormEngine) renderiza los LookUps apuntando al array cacheado de Dominios sin conflictos.
- **MUST NOT:** Ninguna columna huérfana referente a `id_dominio_padre` sobrevive dentro del objeto de esquema de la tabla de Sub-Dominios base.
