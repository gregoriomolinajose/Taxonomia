# Retrospective: S23.1 Admin SDK Connector

## Summary
- **Story ID:** S23.1
- **Dates:** 2026-04-05
- **Time:** Estimated XS (~2h), Actual XS.

## What went well / What to improve
- **Well:** La integración del backend para consumir el entorno API REST de Google usando OAuth delegativo (`executeAs: "USER_DEPLOYING"`) resolvió los problemas de permisos nativos eficientemente. El haber incorporado un `CacheService` backend in-situ protegió la disponibilidad de todo el cluster de Google Workspace.
- **Improve:** Durante el diseño de la Épica no habíamos visibilizado lo crítica que es la arquitectura de *Debouncing* para imágenes concurrentes. En la planificación inicial (Briefing), debería evaluarse la tasa de límite de red siempre que se toque el AdminDirectory u otros 'Advanced Services'.

## Heutagogical Checkpoint
1. **What did you learn?** Usar una primitiva de Javascript pura como un Objeto `Map` con valor de bandera `'pending'` funciona idénticamente a un patrón "Mutex" (Mutual Exclusion) protegiendo el frontend de asfixiar la red sin necesidad de importar grandes librerías de estados o Promesas atómicas.
2. **What would you change about the process?** Realizar un barrido de las cuotas del entorno Google Cloud Platform (GCP) *antes* del diseño arquitectural para blindarlo tempranamente.
3. **Are there improvements for the framework?** Deberíamos documentar como política del proyecto el patrón de interceptar respuestas vacías de red en variables explícitas tipo string como `'null'` en el Caché del servidor, para esquivar los falsy-traps lógicos de JS y así ahorrar llamadas o loops infinitos (el *Muda Prevention*).
4. **What are you more capable of now?** Extraer perfiles corporativos completos del Enterprise Workspace cuidando activamente parámetros de optimización asíncrona mediante fronteras duales (Caché en Cliente + Caché en Servidor).

## Improvements Applied
- Backend recubierto contra iteraciones fantasma (respuestas `null` persistidas 6 horas en Servidor). Frontend asegurado contra sobrecargas por renderizado en cascada (Mutex con fallbacks inyectados síncronamente).

## Patterns Added/Reinforced
- **Add:** `[technical] Usar el String 'pending' en Maps frontends para debouncing pasivo de invocaciones concurrentes.`
- **Reinforce (1):** `[architectural] Caching de Nulls contra Falsy Traps en Backend.`
