## What went well / What to improve
- **Well**: La iteración se movió rapidísimo gracias a que `esbuild` es la herramienta de menor factor de fricción en la industria de JS para transpilación en el backend. 
- **Improve**: El manejo de errores en el bloque antiguo de minificación no aplicaba "Fail Fast". El pipeline habría desplegado silenciosamente si CleanCSS también fallara. Lo hemos arreglado añadiendo `process.exit(1)`.

## Heutagogical Checkpoint
1. **What did you learn?** Aprendí que la API de `esbuild.transformSync()` es un reemplazo `Drop-In` virtual para cadenas literales, capaz de operar sin configuraciones abstractas (sin rollup configs o webpack configs), permitiendo transiciones hiper-ligeras en orquestadores nativos pre-existentes como `deploy.js`.
2. **What would you change about the process?** Re-checar manualmente los bloqueos asíncronos si en el futuro migramos de `transformSync` a `transform` (basado en Promesas).
3. **Are there improvements for the framework?** Incorporar validación de error handling en `QR` como "Golden Pattern" para cualquier CLI que ejecute `child_process` o APIs de transpilación. (Añadido en QA).
4. **What are you more capable of now?** Podemos escalar agresivamente otros minifiers o loaders sin romper GAS (`<?!= ?>`), sentando las bases para S20.2 y S20.3 donde extraeremos JS también.

## Improvements Applied
- Se añadió un guardrail implícito para fallar la CLI si una dependencia de compilación emite un catch. (Fail-Fast aplicado).
