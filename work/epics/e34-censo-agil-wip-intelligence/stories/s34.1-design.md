# Design S34.1: Schema Relación Persona↔Equipo

## 1. Contexto y Valor
**Problema:** El sistema actual no soporta la relación temporal entre Personas y Equipos porque utiliza campos planos (`lookup`) o directamente reglas rotas. Además, la jerarquía entre Equipo y Producto está invertida.
**Valor:** Corregir el esquema permite aprovechar automáticamente el `Engine_DB` y el modelo de grafos para relacionar (y auditar via SCD-2) el personal asignado a cada equipo, base para luego calcular el WIP real.

## 2. Enfoque 
Vamos a intervenir principalmente el motor relacional configurado en `Schema_Engine.gs`:
1. **Reemplazo de campo muerto en Persona:** Mover de `lookupTarget: "Equipas"` a tipo `relation`, definiendo `relationType: 'padre'` (Una Persona pertenece a un Equipo ágilmente), y habilitando `isTemporalGraph: true` con edge `PERSONA_EQUIPO`.
2. **Corrección topológica de la Asignación del Equipo:** En lugar de enlazar el Equipo a un `Producto`, se enlazará a `Grupo_Productos` definiendo `relationType: 'padre'`, `isTemporalGraph: true` e `graphEdgeType: "GRUPO_PRODUCTO_EQUIPO"`. Esto responde a que los equipos atienden flujos de valor/modelos de negocio compartidos.
3. **Subgrid Inverso en Equipo:** Agregar el campo hijo (`Personas_Asignadas`) indicando que el objeto remoto es `Persona` a través del mismo edge.
4. **Segregación de Jerarquía Orgánica:** Validar que la relación recursiva de Recursos Humanos (`Persona.lider_directo`) utilice un `graphEdgeType` explícito (ej. `PERSONA_LIDER_DIRECTO`) para que el engine del grafo no la mezcle con la asignación a Equipos Ágiles.
5. **Estandarización de FIELD_TEMPLATES:** Se completará el Blueprint V4 en Equipo agregando `...FIELD_TEMPLATES.ESTADO_FIELD()` para habilitar soft deletes; y en Persona inyectando `...FIELD_TEMPLATES.SYSTEM_FIELDS()` y `...FIELD_TEMPLATES.ESTADO_FIELD()`.
6. **Reactivación Test de Contrato y E2E:** 
   - En `__tests__/Equipo_CRUD_Fix.test.js` actualizaremos los asserts para el cumplimiento V4 Blueprint asegurando el `primaryKey`, los types exactos de las relaciones, y el estatus soft delete de los Equipos gobernados por Grupos.
   - En `__tests__/e2e/topology-strict.spec.js` agregaremos la cobertura E2E para las profundas jerarquías Ágiles (UN → Portafolio → GP → Equipo → Persona) y Tradicionales (UN → Portafolio → GP → Producto) para certificar integridad.

Para refactoring menor buscaremos si `Controller_Lookups.gs` expone `getEquiposOptions` de manera incorrecta.

## 3. Ejemplos Concretos (Especificación)
El campo `equipo` en `Persona` pasará de:
```javascript
{ name: "equipo", type: "lookup", lookupTarget: "Equipas", label: "Equipo Asignado" }
```
A usar el engine de Grafos:
```javascript
{ 
  name: "id_equipo", 
  type: "relation", 
  label: "Equipo Asignado", 
  relationType: "padre",
  isTemporalGraph: true,
  graphEdgeType: "PERSONA_EQUIPO" 
}
```

El subgrid `personas` en `Equipo`:
```javascript
{ 
  name: "personas_asignadas", 
  type: "relation", 
  relationType: "hijo", 
  childSchema: "Persona",
  isTemporalGraph: true,
  graphEdgeType: "PERSONA_EQUIPO",
  label: "Integrantes",
  virtual: true // No persiste directo en DB_Equipo
}
```

## 4. Criterios de Aceptación Modificados

- **MUST:** Las definiciones de Schema deben incluir estrictamente `isTemporalGraph: true`.
- **MUST:** La propiedad `Equipo.id_producto` debe decir `relationType: 'padre'`.
- **MUST:** Ejecutar NPM TEST y validar que los tests de integridad de `Equipo_CRUD_Fix.test.js` pasan correctamente.
- **MUST NOT:** Modificar el motor interno `Engine_DB.gs` (cualquier fix aquí debe considerarse de alto análisis, y esta historia trata sólo del Schema por ahora).
