# Retrospectiva: S37.4 (Workspace Directory Wrapper)

## Contexto Semántico
Se implementó y conectó el wrapper `resolverDirectorioWorkspace` que orquesta la API de Directorio Activo de Google (Admin SDK) con nuestro cliente Front-End (SPA) para autocompletar e hidratar perfiles demográficos.

## Ejecución Lógica
1. **Promisificación Estricta:** Las llamadas de negocio no tocan `google.script.run` directamente en la UI. Todo se acanaló vía `window.DataAPI.call()`, lo que activa la defensa de "Offline Mocks" necesaria para correr la suite de Vitest local.
2. **Hidratación Resiliente (H10):** Se eliminaron las mutaciones sucias de DOM (`.value`). Ahora el formulario inspecciona las firmas de los Componentes Web y utiliza el protocolo interno de eventos u `Schema_Utils.triggerNativeInject`.
3. **Safe Truthiness:** Se mitigaron falsos vacíos numéricos garantizando paridad en la lógica de JavaScript frente a Ceros literales u objetos vacíos.
4. **Collision Pre-Check:** El Wrapper ahora consulta el datastore reactivo al hidratar, si un empleado o email rescatado de LDAP ya se encuentra insertado en memoria, bota un Warning de colisión sin congelar la pantalla.

## Aprendizajes Duros
- La inyección asincrónica a una SPA debe respetar obligatoriamente el ciclo de vida de Single-Direction Data Flow. Intentar brincarse al Framework provocó pérdida de *State*. Resolverlo emitiendo un `<CustomEvent>` sintético devuelve el control a los detectores MVC en lugar del DOM nativo.
- Las abstracciones tempranas de `DataAPI` nos salvaron de un infierno de testing para las promesas.

## Status Final
Aprobado. El autollenado Zero-Touch es funcional y totalmente Mockeable.
