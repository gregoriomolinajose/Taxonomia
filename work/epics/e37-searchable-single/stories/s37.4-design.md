# Diseño: S37.4 (Wrapper de Workspace Directory)

## Propósito Objetivo
Construir una pasarela de comunicación (Wrapper) fluida entre el Motor de Interfaces Dinámicas en el Frontend (`UI_FormDependencies.client.js`) y la API de Google Admin SDK en el Backend (`Engine_Workspace.gs`). El propósito central es poblar automáticamente todos los datos demográficos y estructurales de una "Persona" (avatar, departamento, centro de costos, cargo, líder, herradura y número de empleado) tan pronto insertan su email corporativo válido.

## Conceptos Core

### 1. The Trigger (`triggers_workspace_resolve: true`)
Todo campo en `Schema_Engine.gs` que contenga esta bandera (Ej. `email`), actuará de detonante en el evento `ionBlur` o `ionChange`. El Framework escuchará esto nativamente en el Gestor de Dependencias.

### 2. The Abstraction Layer (DataAPI Client)
El Frontend no debe invocar `google.script.run` directamente por temas de inyección de testabilidad local (vitest). En su lugar, el Wrapper inyectará la llamada a través del enrutador central de llamadas usando promesas en JS:
`window.DataAPI.call('resolverDirectorioWorkspace', email).then(...)`

### 3. Asignación Bi-Direccional Nodal (Reactivity)
Cuando el Backend devuelve el Payload (DTO), el Wrapper iterará sobre la llave y valor, buscará el `ion-input` correspondiente usando `document.querySelector(input[name="${key}"])` y le forzará el `.value`. 

## Flow Arquitectónico

1. User ingresa `"luis.perez@dominio.com"` en *[name="email"]*.
2. El debounce del Frontend evalúa que el Regex es válido.
3. Se dispara Loading Visual (Spinner inline).
4. **DataAPI** llama a **Engine_Workspace.resolverDirectorioWorkspace**.
5. Back-end hace Lookup en Google LDAP Server public domain info.
6. Mapeo Defensivo encapsula el resultado en Payload JSON.
7. Wrapper de Interfaz desenrolla el JSON y asigna a las cajas correspondientes de la vista.
8. Se gatillan eventos pasivos para validar los campos recién rellenados.

## Acceptance Criteria
- **MUST**: Operar de manera asíncrona (Async/Await) sin congelar el Hilo de UI principal del Navegador.
- **MUST**: Funcionar como un Graceful Degradation. Si la API contesta { __status: "DISABLED" } o un error HTTP, silencia y permite al usuario llenar los datos manualmente.
- **MUST NOT**: Modificar directamente lógica dentro de `UI_FormSubmitter`, manteniendo la responsabilidad separada en `UI_FormDependencies`.
