# S37.7 Design: Refactorización de IoC en UI_DataGrid

## Descripción del Problema
Actualmente, la orquestación del DataGrid depende del antipatrón de _Evaluación Global de Strings_ (`_invoke`). Esto surgió porque históricamente esta vista corría bajo limitaciones del runtime de Google Apps Script donde compilar handlers al vuelo requería strings en el template HTML. Ahora, bajo la nueva directiva _V4 Node Purity_ de la Taxonomía, la interfaz DOM es construida íntegramente de manera declarativa con componentes JS (`document.createElement`).

## Solución Propuesta (IoC)
Se propone desterrar el acceso sucio al window scope aplicando formalmente _Inversion of Control_ mediante Dependency Injection.
El `DataView_UI` (que maneja el Scope Orquestador) pasará explícitamente sus métodos por referencia como callbacks.
El `UI_DataGrid` ejecutará estos _callbacks_ chequeando que respeten el tipo `function`.

### Contract Update
```javascript
// ANTIGUO PAYLOAD DE CONFIG
{
    onSort: 'window.DataViewEngine._onSort', // Fragilidad a ofusación, scope externo forzado
}

// NUEVO PAYLOAD DE CONFIG (Inyección explícita)
{
    onSort: (colId) => _onSort(colId),
    onDelete: window.AppUI.confirmDelete // Resolviendo las deps globales en el caller, no callee
}
```

## Beneficios
1. Testeabilidad nativa.
2. Inmunidad a minificadores como Terser o Esbuild, y a pérdida condicional de scope de _window_.
3. Cohesión Arquitectónica con `UI_Factory` y modales de la suite E37.
