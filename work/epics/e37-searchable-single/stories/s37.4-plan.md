# Plan de Implementación: S37.4 (Workspace Directory Wrapper)

## Contexto de la Historia
Actualmente el `Schema_Engine.gs` marca el campo `email` y `nombre` con `triggers_workspace_resolve: true`. Este plan detalla los pasos para completar la hidratación o autollenado transaccional y visual del frontend al momento en el que el ecosistema emite este evento.

## Tareas Atómicas

### Tarea 1: API Router Registration
- **Modificar:** `src/API_Universal.gs` o `src/Controller_Action.gs` (Dependiendo la capa de enrutamiento).
- **Descripción:** Registrar un *endpoint* o case para direccionar peticiones RPC hacia la función nativa `resolverDirectorioWorkspace(query)` encapsulada en `Engine_Workspace.gs`.

### Tarea 2: Dependency Engine Resolver
- **Modificar:** `src/UI_FormDependencies.client.js`
- **Descripción:** Cuando se dispare el trigger, envolver la llamada al backend usando la asincronía de `window.DataAPI.call()`. Se debe añadir soporte para un icono o toast nativo de "Loading" que informe al usuario que la plataforma está "Buscando en el Directorio Corporativo...".

### Tarea 3: Reactividad de Inputs y Sincronización Estructural (Hydration)
- **Modificar:** `src/UI_FormDependencies.client.js`
- **Descripción:** Dado el payload exitoso (`dto`), iterar sobre las llaves devueltas y aplicar un "Setter Defensivo". En lugar de forzar `.value` en crudo, verificar si el elemento localizado es un WebComponent enriquecido que requiera métodos dedicados. En inputs estándar, usar el assignment setter y despachar sintéticamente eventos `ionChange`/`input` que detonen validaciones visuales sin quebrar el estado interno del Componente SPA.

### Tarea 4: Abortos y Modos Degradados
- **Modificar:** `src/UI_FormDependencies.client.js`
- **Descripción:** Manejar cortadores de circuito. Si el DTO trae `__status: "DISABLED"` (porque lo estamos corriendo en CI/CD local test) o si falla la conexión, fallar silenciosamente imprimiendo un warning en la consola y dejar que el usuario escriba sus datos a mano sin bloquearle el form.
