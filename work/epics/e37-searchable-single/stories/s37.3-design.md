# Story Design S37.3: Prevención de Colisiones de Identidad

## Qué & Por Qué
- **Problema**: Algunos usuarios logran bypassear la UI y guardar una misma "Persona" o "Entidad" varias veces, cambiando ligeras variaciones como mayúsculas, pero el sistema lo mapea como UUIDs diferentes. 
- **Valor**: La integridad de la información. Un sistema de gobernanza no puede permitirse "gemelos malvados", ya que arruina el Motor de ABAC, los reportes analíticos y la experiencia en listados cruzados. 

## Enfoque de Solución
Validación Pre-Commit en el Frontend (Zero-Latency). Como el explorador ya tiene la base de datos descargada en la RAM (`window.DataStore`), evaluamos O(1) antes de enviar el Payload al servidor, protegiendo al backend de hacer búsquedas lineales de I/O por fuerza bruta.

### Componentes Afectados
- `UI_FormSubmitter.client.js` [MODIFICADO]: Actúa como muro perimetral. Antes del request asíncrono, chequea contra `window.DataStore` las columnas marcadas como `unique: true` o `triggers_workspace_resolve: true`.

## Ejemplos Concretos (AC Tests)

```javascript
const uniquenessFields = this.fields.filter(f => f.triggers_workspace_resolve || f.unique);
const hit = liveData.find(r => r.estado !== 'Eliminado' && String(r[field.name]).toLowerCase() === String(valToCheck).toLowerCase());
if (hit && String(hit[pkField]) !== String(this._internalRetryId)) {
    // Abort flow!
    this._showToast(`⚠️ Colisión Detectada: Ya existe un registro vigente con ${field.label} = ${valToCheck}.`);
}
```

## Acceptance Criteria
- **MUST**: Los esquemas con `unique: true` abortan la transacción del formulario antes del hit de red.
- **MUST**: El usuario recibe feedback rojo visual (Toast) comunicando la interdicción.
- **MUST NOT**: Realizar lecturas lineales en memoria backend de GAS que rebasen los límites de la cuota de ejecución.
