# Story Design S37.5: Módulo de Importación Bulk Sheets (ETL)

## Qué & Por Qué
- **Problema**: La captura manual de directorios corporativos extensos (cientos de registros) no es viable. Las técnicas de bulk insert tradicionales en Google Workspace superan el límite de 6 minutos crasheando la sincronización en nube.
- **Valor**: La carga por lotes (Chunked) permite al administrador integrar grandes volúmenes de datos superando las limitantes de GAS de forma resiliente con feedback visual contínuo.

## Enfoque de Solución
El componente de importación usará la UI del Cliente para dividir el CSV/Petición en "Chunks" o lotes de hasta 50 elementos.
Cada lote enviará un Payload y esperará resolución (Batching Client-Side).
Se creará un proxy Backend ETL sobre el nuevo puerto API que consumirá `Engine_DB.upsertBatch()`

### Componentes Afectados
- `UI_DataView_Toolbar.client.js` [MODIFICAR]: Añadir Input File y Modal de Progreso.
- `API_Universal.gs` [MODIFICAR]: Enrutador a la funcionalidad de Upsert Batch.

## Ejemplos Concretos (AC Tests)

```javascript
// El cliente dispara al Engine Batch a través del proxy orquestador DataAPI:
const chunk = [
  { nombre: "Usuario Test 1", email: "user1@corp.com" },
  // ... 49 más
];

// Invocación segura Mockeable:
window.DataAPI.call('bulkInsert', 'Persona', chunk)
  .then((res) => {
     UI_ProgressBus.update("chunk 1/10 procesado");
  })
  .catch((err) => console.error(err));
```

## Acceptance Criteria
- **MUST**: El Frontend divide arreglos grandes en partes pequeñas antes de enviarlas al servidor.
- **MUST**: Se despliega una interfaz de barra de progreso asíncrona hasta terminar el lote.
- **MUST**: Utiliza `Engine_DB.upsertBatch` internamente sin modificar la abstracción.
- **MUST NOT**: Utilizar bucles iterativos For() unitarios llamando repetidamente al DB Writer en Backend. Obligatoriamente será Unit of Work en Backend.
