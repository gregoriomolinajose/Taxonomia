# Story S25.2 Retrospective: Mobile First Drawer & Workspace Hydration & Graph Transition

## Summary
- **Story ID:** S25.2
- **Dates:** 2026-04-06
- **Estimated Time:** 1 day
- **Actual Time:** 2 days

## What went well / what to improve
- **Went well:** The integration of the Google Workspace API using the `FormHydrated` event was clean and avoided Angular-style two-way binding loops. The zero-UI-latency paradigm was maintained gracefully.
- **To improve:** The architectural transitions for SCD-2 graphs require better holistic analysis of the Data Layer. We caught a critical regression in `Engine_DB.js` during the final code review where it attempted to save the entire edge payload atop base nodes instead of delegating to the `graphEntity`.

## Heutagogical Checkpoint

1. **What did you learn?**
Learned that adapting legacy flat structures into normalized temporal edges requires ensuring the write-path (upsertBatch) respects the new table target (`graphEntity`) rather than persisting legacy parent references in the base nodes. I learned about the importance of explicitly routing batch operations differently between native nodes and temporal associative entities.

2. **What would you change about the process?**
We should have audited the persistence layer (`Engine_DB.js`) side-by-side with the schema definitions (`Schema_Engine.gs`) before merging fields that lost `foreignKey`. We assumed `ETL_Runner.gs` behavior applied to in-app writes automatically.

3. **Are there improvements for the framework?**
Yes, we should add an automated validation gate in the `API_Universal_Router` or `Schema_Engine` test suites (`__tests__/`) to verify that queries labeled as `isTemporalGraph: true` never hit their `targetEntity` during associative saves.

4. **What are you more capable of now?**
Navigating declarative UI schemas to orchestrate complex DB Engine writes across multiple Google Sheets tabs simultaneously without locking up the UI, keeping robust architecture using plain JavaScript and Google Apps Script boundaries.

## Improvements Applied
- Updated `Engine_DB.js` to correctly route SCD-2 operations to `f.graphEntity` dynamically, and correctly map `tipo_relacion` to the field name instead of a hardcoded "Militar_Directa".
- Fixed tests in `__tests__/Persona_CRUD.test.js` to align with the new V4 Persona schema structure (`numero_empleado` as primary key).

## Patterns added/reinforced
- **Reinforced:** Extracted monolithic components into declarative Pub/Sub workflows.
- **Added:** PAT-G-027 ("When migrating relations to Temporal DAGs (SCD-2), ensure the DB Engine's write-path targets the graphEntity explicitly and injects the relation type dynamically, preventing cross-contamination into entity nodes.")
