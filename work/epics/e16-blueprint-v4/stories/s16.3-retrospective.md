# S16.3 Retrospective: DataGrid Minimalism & Layout Extraction

## Verification
- **Test Strategy:** Inspección manual del `DataView_UI` con simulación AST tras separar las barras de filtros a su propio contenedor abstracto y vaciar logica de control hacia iteradores "tontos". No hubo regresiones en filtros multi-columna.
- **Failures Found:** Ninguno durante su desarrollo base; sin embargo, en QA el Search Input temporalmente perdió foco en el tipado rápido, lo que obligó a delegar la gestión del "re-pintado" al DOM VDOM Diffing en vez de un re-render global (`innerHTML`).

## Learnings
1. **Delegation Pays Off:** Extraer el Search Bar a módulos apartes simplificó el motor de Virtual DOM del DataGrid en casi 200 LOC, validando la regla de SRP en Componentes visuales.
2. **Event Listeners in Loop:** Se reafirmó la importancia de adherir listeners de interacción (Clicks) en el Parent Node (Event Delegation) en vez de hacerlo en un `map()` de cada fila de datos (Previene Memory Leaks sustancialmente).

## Next Steps / Process Changes
- Aplicar este patrón DataGrid Minimalist a otras tablas legacy de la plataforma (como SubgridBuilder) que todavía operan concatenando en ES5 puro para re-pintado reactivo.
- Añadir alerta al framework para no embeber `input` dinámicos dentro de elementos que redibujan sobre el `onInput` event (React Pattern).
