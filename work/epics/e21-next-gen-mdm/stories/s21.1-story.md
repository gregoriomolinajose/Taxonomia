# Story S21.1: Optimistic Locking Pipeline

## User Story

**As an** end-user / data-entry operator  
**I want** the system to prevent me from overriding data if another user modified it while I was looking at it  
**So that** we avoid corrupting or silently losing information during concurrent operations.

## Acceptance Criteria

### Backward Compatibility
- **Given** an old entity without a `metadata.version`
- **When** it is loaded into the frontend and updated
- **Then** the engine should initialize its version safely starting at `1`.

### Concurrency Block
- **Given** two users (A and B) load Entity X (version `1`)
- **When** User A saves, Entity X becomes version `2`
- **Then** if User B attempts to save their copy (with version `1`), the backend rejects it with an explicit Error (`OptimisticLockError`).

### Graceful Degradation (UI)
- **Given** the backend throws an `OptimisticLockError`
- **When** `FormSubmitter` catches it
- **Then** the modal must not close, and an explanatory warning (`toast` or alert) must notify the user of the collision.
