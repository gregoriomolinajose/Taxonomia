## What Went Well

1. **Dispatcher pattern (S31.10)** — El refactor de renderComposerTab en 3 ramas fue limpio y permitio que el entity composer original quedara 100% intacto.
2. **_TEMPLATE_CATALOGUE como espejo estatico** — Aunque crea drift risk, la decision de no hacer google.script.run async simplifica el init enormemente. La alternativa tenia un costo disproportionado.
3. **Architecture Review aplicado inmediatamente** — Detectar R1+R2 en el mismo ciclo y aplicarlos antes del next-cycle es la practica correcta. Costo: 15 min. Beneficio: -35 lineas de deuda.
4. **AST validation via acorn** — Gate que atrapó 2 stray braces y 1 self-call recursivos introducidos por automatizacion de PowerShell.

---

## What To Improve

1. **Refactors automatizados en HTML+JS embebido son fragiles** — ForEach PowerShell matcheo dentro de funciones recien creadas (self-call en _activeTemplate). En el futuro, usar reemplazos mas acotados (line-range especifico) en lugar de patrones globales.
2. **_copyTabContent debia invocarse desde el inicio** — Las funciones composerTemplateCopy y composerBehaviorCopy debieron escribirse directamente delegando al helper. La duplicacion fue un error de diseño en la implementacion inicial, no detectado hasta el AR.

## Heutagogical Checkpoint

**Q: ¿Que aprendiste?**
En GAS single-file, cada patron de comportamiento (clipboard, lookup, render) debe tener exactamente UN punto de implementacion desde el inicio. El IIFE protege el scope pero no elimina la tentacion de duplicar logica al agregar nuevos contextos.

**Q: ¿Que cambias del proceso?**
La validacion AST debe correr antes de commitear, no despues. El CI actual la corre en deploy.js, pero para refactors automatizados debe ejecutarse en el mismo script de refactor.

**Q: ¿Hay mejoras para el framework?**
Agregar un guardrail: "Si el archivo modificado es HTML con JS embebido, ejecutar AST validation ANTES de git add". Candidato para un pre-commit hook.

**Q: ¿En que eres mas capaz ahora?**
Diseñar Composers multi-contexto en GAS sin frameworks: sidebar DOM-native, panel DOM-native, state en closure IIFE, CODE EXPORT paramétrico. El patron es replicable para cualquier nuevo tab del Studio.
