# S31.9 · Server-Side Auth Guard en API_Admin.gs

## Contexto

Durante el Quality Review de E31, la observación de seguridad identificó que
`API_Admin.gs` expone las funciones `runSchemaReconcile()` y `getSchemaProvisioningStatus()`
como GAS globales sin ningún guard de autenticación server-side.

El modelo de seguridad actual depende **exclusivamente** del guard client-side (ABAC):
el botón "Sistema" del popover solo es visible para `SUPER_ADMIN`. Sin embargo, cualquier
usuario autenticado en el script puede llamar `google.script.run.runSchemaReconcile()`
desde la consola del navegador y ejecutar una reconciliación completa del schema de BD.

## Story Statement

**Como** arquitecto de seguridad de la plataforma,  
**quiero** que las funciones admin de `API_Admin.gs` validen el rol del invocador
en el servidor antes de ejecutar operaciones privilegiadas,  
**para que** el acceso a governance de schema esté protegido por defensa en profundidad
(client-side ABAC + server-side role check) y no sea bypasseable desde la consola.

## Criterios de Aceptación

- [ ] `runSchemaReconcile()` verifica que el email del invocador pertenece a la lista
  `SUPER_ADMIN_EMAILS` almacenada en `Script Properties` antes de ejecutar.
- [ ] `getSchemaProvisioningStatus()` aplica el mismo guard.
- [ ] `ensureEntityProvisioned()` aplica el mismo guard.
- [ ] Si el invocador no está autorizado, la función lanza `Error('Unauthorized: SUPER_ADMIN required')`.
- [ ] El error es capturado correctamente por `withFailureHandler` en el cliente y se muestra un toast de error.
- [ ] La lista `SUPER_ADMIN_EMAILS` se administra via GAS Script Properties (no hardcoded).
- [ ] Existe documentación en el README del proyecto sobre cómo configurar `SUPER_ADMIN_EMAILS`.

## Notas de Diseño

### Guard Helper (DRY)

```js
// API_Admin.gs

/**
 * Server-side SUPER_ADMIN guard.
 * Throws if the active user is not in the SUPER_ADMIN_EMAILS list.
 * List stored in Script Properties: SUPER_ADMIN_EMAILS (comma-separated).
 */
function _requireSuperAdmin() {
  const email = Session.getActiveUser().getEmail();
  const prop  = PropertiesService.getScriptProperties()
                  .getProperty('SUPER_ADMIN_EMAILS') || '';
  const admins = prop.split(',').map(e => e.trim().toLowerCase());
  if (!admins.includes(email.toLowerCase())) {
    Logger.log('[API_Admin] Unauthorized access attempt by: ' + email);
    throw new Error('Unauthorized: SUPER_ADMIN required');
  }
}
```

### Uso en cada endpoint

```js
function runSchemaReconcile() {
  _requireSuperAdmin();  // ← guard first
  return reconcileAll(SpreadsheetApp.getActiveSpreadsheet());
}
```

### Configuración de Script Properties

En GAS > Proyecto > Configuración del proyecto > Propiedades del script:
```
SUPER_ADMIN_EMAILS = "admin@empresa.com,dev@empresa.com"
```

### Fallback durante desarrollo

Si `SUPER_ADMIN_EMAILS` está vacía o no configurada, el guard debe loguear una
advertencia pero **NO bloquear** (para evitar lockout durante setup inicial):

```js
if (!prop) {
  Logger.log('[API_Admin] WARNING: SUPER_ADMIN_EMAILS not configured. Guard bypassed.');
  return; // fail-open during setup
}
```

## Archivos Afectados

- `src/API_Admin.gs` — agregar `_requireSuperAdmin()` + llamarlo en cada endpoint
- `README.md` o `docs/setup.md` — documentar configuración de `SUPER_ADMIN_EMAILS`
