# Story S12.2: DataGrid & Routing Abstraction

## User Story (Connextra)
**As a** Arquitecto SPA
**I want to** aislar la lógica de enrutamiento y la factoría de generación de tablas desde `JS_Core.html` y `DataView_UI.html`
**So that** los módulos adquieran pureza funcional, se reduzca dramáticamente el número de líneas (evitando *Fat Scripts*) y el enrutamiento opere independientemente de la carga central.

## Acceptance Criteria (Gherkin)

**Scenario 1: Routing Separation**
- **Given** the current State Machine and URL sync logic in `JS_Core.html`
- **When** the routing engine is permanently extracted to `UI_Router.html`
- **Then** `JS_Core` must initialize routing without directly handling `window.history` APIs.
- **And** page transitions (View -> Form -> Dash) work seamlessly as before.

**Scenario 2: DataGrid Abstraction**
- **Given** building HTML tabular grids programmatically inside `DataView_UI.html`
- **When** the DOM grid builder logic is isolated to `UI_DataGrid.html` (o equivalent)
- **Then** the `DataView_UI` simply orchestrates and delegates DOM nodes polymorphically.
- **And** the data display matches exactly the previous CSS/UI Design System Tokens.
