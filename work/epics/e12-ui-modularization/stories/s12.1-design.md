# Story Design: S12.1 - FormEngine Splitting

## What & Why
**Problem:** `FormEngine_UI.html` (1700+ LOC) centralizes orchestration, config parsing, DOM creation, and HTTP interactions. It's a monolith violating SRP.
**Value:** Moving DOM nodes construction to a pure factory (`FormBuilder_Inputs`) improves maintainability to O(1) and shields the `FormRenderer` orchestration from bloat.

## Approach & Architecture
- **FormBuilder_Inputs.html:** Exposes `window.UI_Factory` for generating pure `<ion-item>` sub-trees without side-effects.
- **FormRenderer_UI.html:** Only iterates `APP_SCHEMAS`, calls the factory, and manages `save()`.
- **FormValidators.html:** Isolates business logic rules.
- **Value Preservation:** `LocalEventBus` must be injected into the Builder to preserve hierarchical SCD-2 bindings.

## Examples
```javascript
// New invocation from Renderer:
const node = window.UI_Factory.buildInput(fieldSchema, currentValue, localEventBus);
container.appendChild(node);
```
