# Story S12.1: FormEngine Splitting

## User Story (Connextra)
**As a** Arquitecto SPA y Desarrollador Frontend
**I want to** aislar la lógica nativa del DOM (`document.createElement`) de la lógica de orquestación reactiva y persistencia
**So that** los componentes (Selects, Textareas) sean fábricas testables puras y nuestro motor rinda fluidamente bajo presión sin fatigar la RAM del cliente, manteniéndose libre de "Fat File Smells" superiores a 800 líneas.

## Acceptance Criteria (Gherkin)

**Scenario 1: Modularization boundaries respect SRP**
- **Given** I inspect the `FormEngine_UI.html` equivalent logic
- **When** the DataView transitions to a Form View
- **Then** the primary `renderForm` file only orchestrates state and calls an external dependency (`FormBuilder_Inputs`) to resolve exact `<ion-item>` widgets.
- **And** the renderer size is strictly under 800 LOC.

**Scenario 2: Data Binding Resilience (Zero Regressions)**
- **Given** the form contains native dependencies (SCD temporal dates, cascading Selects)
- **When** I compile the split `FormBuilder_Inputs.html` widgets independently
- **Then** all widgets continue to publish and subscribe to the `LocalEventBus` natively.
- **And** typing inside nested fields fires changes down exactly identically to the E11 monolith.

**Scenario 3: Universal Fallback Injection**
- **Given** specific missing nodes inside the Builder logic
- **When** a widget requires validation rules not present in its atom scope
- **Then** the `FormValidators.html` kicks in seamlessly at Save-time resolving data integrity without crashing the Renderer.
